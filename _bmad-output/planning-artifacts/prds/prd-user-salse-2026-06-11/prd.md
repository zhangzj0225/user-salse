# PRD: 足球舆情系统 — 多级分销小程序

## 0. Document Purpose

本文档面向开发团队（前端小程序 + 后端 + 管理员），定义「足球舆情系统」多级分销小程序的完整功能需求。文档以 Glossary 锚定核心术语，Features 章节按功能模块组织并嵌套全局编号的 FR，所有推断性内容以 `[ASSUMPTION]` 标记并在 §9 索引。

上游输入：`足球舆情系统销售措施.pdf`（业务规则文档）。

---

## 1. Vision

本系统是一个基于微信小程序的**多级分销管理工具**，核心解决三个问题：

1. **邀请链传播**：代理可发展经销商、也可发展代理；经销商可发展自己的经销商。所有关系通过邀请码明文传播，系统记录完整的上下级链条。
2. **佣金自动计算**：根据角色和业务场景（发展下线、自销账号、团队业绩），系统自动计算各级佣金。
3. **团队与收益可视**：每个用户可查看自己的团队结构、收益明细；管理员可在后台管理用户、审核提现工单。

系统定位为**半公开运营工具**——有公开入口但主要通过邀请码传播，面向足球舆情行业的代理/经销商体系。

## 2. Target User

### 2.1 角色定义

| 角色 | 说明 |
|------|------|
| **普通用户** | 未付费准入的注册用户，可购买账号、推荐新用户获得返佣 |
| **经销商** | 支付 5000 元准入费，获得 11 个账号，可发展下线经销商/代理，享受团队业绩提成 |
| **代理** | 支付 10000 元准入费，获得 22 个账号，可发展下线经销商/代理，享受团队业绩提成（比例高于经销商） |
| **管理员** | 后台管理角色，负责用户审核、工单处理、系统配置 |

### 2.2 Jobs To Be Done

- 作为代理，我需要通过邀请码发展经销商和下级代理，并实时看到我的团队规模和佣金收益
- 作为经销商，我需要通过邀请码发展下级经销商，销售账号赚取佣金
- 作为普通用户，我需要注册后获得邀请码，推荐他人购买账号获得返佣
- 作为管理员，我需要审核用户准入、处理提现工单、查看系统运营数据

### 2.3 Non-Users (v1)

- 不面向未注册的游客（所有功能需登录后使用）
- 不面向非微信环境（仅微信小程序）

### 2.4 Key User Journeys

- **UJ-1. 新用户通过代理的"发展代理"邀请码注册。**
  - **Persona + context:** 张先生收到朋友（代理）发来的邀请码，该邀请码是朋友在"发展代理"模式下生成的。
  - **Entry state:** 未注册，通过朋友分享的小程序卡片进入。
  - **Path:** 打开小程序 → 微信授权登录 → 绑定手机号 → 输入邀请码 → 系统解析邀请码，识别目标角色为"代理"（用户无需手动选择）→ 提交准入申请 → 线下支付 10000 元准入费 → 管理员后台确认收款 → 账户激活为代理，获得 22 个账号额度。系统记账记录上级的首次奖励 5500 元（待提现）。
  - **Climax:** 管理员确认后，系统自动建立与邀请人的上下级关系，双方可见；上级收益余额中新增 5500 元记账。
  - **Resolution:** 张先生成为代理，拥有自己的邀请码，可继续发展下线。

- **UJ-2. 代理生成"发展经销商"邀请码，邀请王先生注册。**
  - **Persona + context:** 李代理想发展经销商下线。
  - **Entry state:** 李代理已登录，在"邀请好友"页面。
  - **Path:** 李代理选择"发展经销商"模式 → 系统生成包含经销商目标角色信息的邀请码 → 李代理分享给王先生 → 王先生注册时填写该邀请码 → 系统解析邀请码，自动确定角色为"经销商" → 提交准入申请 → 线下支付 5000 元 → 管理员确认 → 系统记账李代理的首次奖励 2750 元。王先生后续每销售一个账号，系统自动记账李代理的账号销售提成 133.2 元。
  - **Climax:** 李代理在"我的收益"中看到 2750 元记账入账。
  - **Resolution:** 王先生成为李代理的下级经销商，关系链记录完成。

- **UJ-3. 用户查看自己的团队链条。**
  - **Persona + context:** 赵代理想了解自己发展了哪些人，以及这些人又发展了谁。
  - **Entry state:** 已登录。
  - **Path:** 进入"我的团队" → 查看树形/列表结构 → 看到直接下级（代理/经销商）→ 点击某下级可展开其下级 → 逐层追溯完整链条。
  - **Climax:** 清晰看到自己的团队规模和层级结构。
  - **Resolution:** 赵代理了解团队全貌，可针对性指导或激励下级。

- **UJ-4. 管理员处理提现工单。**
  - **Persona + context:** 管理员小王收到代理的提现申请。
  - **Entry state:** 已登录后台管理系统。
  - **Path:** 进入"工单管理" → 查看待处理提现工单列表 → 点击某工单查看详情（申请人、金额、佣金来源明细）→ 确认无误后点击"已打款" → 系统更新工单状态 → 申请人端显示"已打款"。
  - **Climax:** 工单状态变更，申请人收到通知。
  - **Resolution:** 提现流程闭环，财务记录可追溯。

## 3. Glossary

- **用户** — 在小程序中完成微信授权和手机号绑定的自然人。每个用户拥有唯一 ID 和个人邀请码。
- **邀请码** — 系统按隐性规则生成的唯一标识字符串（明文），用于建立上下级关系。邀请码由上级在生成时指定目标角色（"发展代理"或"发展经销商"），编码了目标角色信息。新用户注册时填写邀请码，系统自动解析并确定其角色，用户无需手动选择。`[ASSUMPTION: 邀请码按隐性规则生成，包含生成者 ID 和目标角色类型编码]`
- **上级/下级** — 通过邀请码建立的直接推荐关系。A 的邀请码被 B 注册时使用，则 A 是 B 的上级，B 是 A 的下级。
- **团队** — 某用户的所有下级（含间接下级）构成的树形结构。根节点为该用户自身。
- **链条** — 从某用户向上追溯到根节点（无上级的用户）的完整路径，或从某用户向下展开的全部子树。
- **角色** — 用户在分销体系中的身份：普通用户、经销商、代理。角色决定佣金比例和权限。
- **准入费** — 成为经销商（5000 元）或代理（10000 元）需支付的一次性费用。支付后获得对应数量的账号额度。
- **账号** — 足球舆情系统的使用账号。代理获得 22 个，经销商获得 11 个，普通用户 0 个。定价 888 元/个。
- **自销** — 用户直接销售账号给终端客户的行为。
- **首次奖励** — 发展下级（代理或经销商）时，上级获得的一次性佣金，按被发展者准入费的一定比例计算。
- **长期奖励** — 下级团队后续销售业绩中，上级按比例持续获得的佣金。
- **账号销售提成** — 下级销售账号时，上级按比例获得的佣金。
- **工单** — 用户发起的提现申请记录，由管理员后台审核并线下打款后结单。
- **管理员** — 拥有后台管理权限的特殊账号，可管理用户、处理工单、查看运营数据。
- **记账** — 系统对佣金和收益的计量记录。所有佣金计算后以"记账"形式计入用户余额，不代表真实资金已到账。用户需通过提现工单将记账余额转为实际收款。系统不处理真实资金流转，仅做账务记录。

## 4. Features

### 4.1 用户注册与认证

**Description:** 用户通过微信小程序完成注册。流程：微信授权获取 OpenID → 绑定手机号 → 输入邀请码（可选）→ 若邀请码有效，系统自动解析目标角色；若无邀请码，默认为普通用户。注册后系统自动生成个人邀请码。`[ASSUMPTION: 微信授权使用 wx.login + 手机号快速验证组件]`

**Functional Requirements:**

#### FR-1: 微信授权登录
用户打开小程序后，通过微信一键授权登录。系统获取用户 OpenID 和头像昵称。Realizes UJ-1.

**Consequences (testable):**
- 未授权用户无法进入任何功能页面
- 授权成功后自动跳转至首页或注册引导页
- 已授权用户再次打开小程序时自动登录

#### FR-2: 手机号绑定
用户在注册流程中绑定手机号。系统验证手机号唯一性。

**Consequences (testable):**
- 手机号已注册时提示"该手机号已绑定"，不允许重复绑定
- 手机号格式不合法时提示错误
- 绑定成功后用户状态标记为"已激活"

#### FR-3: 邀请码填写与角色解析
用户在注册时可填写邀请码。邀请码为明文字符串，用户手动输入。系统解析邀请码，自动确定用户的目标角色（代理或经销商），用户无需手动选择角色。解析成功后，系统展示目标角色、对应准入费用和权益说明，用户确认后进入注册流程。

**Consequences (testable):**
- 邀请码不存在时提示"邀请码无效，请联系分享者确认"
- 邀请码有效时，系统解析出目标角色并展示：角色名称、准入费用、获得账号数量、佣金规则概要
- 用户确认角色信息后继续注册
- 邀请码有效时建立上下级关系
- 邀请码可不填（用户成为根节点，角色为普通用户）
- 用户不能填写自己的邀请码

#### FR-4: 个人邀请码生成（含目标角色选择）
注册成功后，系统为用户生成基础邀请码。代理/经销商用户在"邀请好友"页面可选择目标角色（"发展代理"或"发展经销商"），系统据此生成包含目标角色信息的邀请码。邀请码需包含防篡改签名，防止伪造角色。普通用户仅生成通用邀请码（推荐新用户购买）。

**Consequences (testable):**
- 邀请码全局唯一，按隐性规则生成，包含防篡改签名
- 代理可生成两种邀请码：发展代理、发展经销商
- 经销商可生成两种邀请码：发展代理、发展经销商
- 普通用户仅生成通用推荐邀请码
- 用户可在"我的"页面查看和复制自己的邀请码
- 邀请码不可修改
- 被篡改的邀请码解析失败，提示"邀请码无效"

### 4.2 角色与准入

**Description:** 用户角色由邀请码确定（代理/经销商）或默认为普通用户。准入费支付为线下方式，由管理员在后台确认。确认后用户角色生效，获得对应账号额度。系统仅做记账，不处理真实资金流转。

**Functional Requirements:**

#### FR-5: 准入申请
用户注册后（角色由邀请码确定或默认为普通用户），若角色为代理/经销商，系统自动生成待审核的准入记录。用户线下完成支付后，由管理员在后台确认。

**Consequences (testable):**
- 通过邀请码注册的用户，角色由邀请码确定，无需手动选择
- 无邀请码注册的用户默认为普通用户
- 已是经销商/代理的用户不可重复申请同角色
- 经销商不可升级为代理（v1 不支持）

#### FR-6: 准入费支付（线下 + 后台确认）
用户提交准入申请后，系统生成待审核记录。用户线下完成支付，管理员在后台确认收款后，用户角色生效。系统自动记账上级的首次奖励（计入余额，待提现）。

**Consequences (testable):**
- 提交申请后用户状态为"待审核"
- 管理员确认后用户角色变更为对应角色
- 管理员确认后系统自动分配账号额度（代理 22 个，经销商 11 个）
- 管理员确认后系统自动记账上级的首次奖励
- 所有佣金以"记账"形式记录，不涉及真实资金流转

**Out of Scope:**
- 在线支付（微信支付）集成

#### FR-7: 账号额度管理
用户可在"我的账号"中查看剩余账号数量和已销售记录。额度耗尽后可向管理员申请补购，管理员在后台审核并追加额度。

**Consequences (testable):**
- 代理初始额度 22 个，经销商 11 个，普通用户 0 个
- 销售账号后额度递减
- 额度为 0 时不可继续销售
- 额度耗尽后显示"申请补购"入口，提交补购申请（指定数量）
- 管理员在后台审核补购申请，确认收款后追加额度
- 补购记录可追溯

### 4.3 邀请链与团队管理

**Description:** 核心功能。系统记录完整的邀请关系链，每个用户可查看自己的上级链条和下级团队树。

**Functional Requirements:**

#### FR-8: 邀请关系记录
系统在用户注册填写邀请码时，记录邀请关系（上级 ID、下级 ID、建立时间）。

**Consequences (testable):**
- 关系数据包含：上级用户 ID、下级用户 ID、邀请码、创建时间
- 关系不可删除或修改
- 支持通过关系表追溯完整链条

#### FR-9: 我的团队（下级树）
用户可查看以自己为根节点的下级团队树。展示直接下级列表，点击可展开间接下级。

**Consequences (testable):**
- 展示每个下级的昵称、角色、注册时间
- 展示每个下级的直接下级数量
- 支持逐层展开，不限层级深度
- 展示团队总人数统计

#### FR-10: 我的上级链条
用户可查看从自己向上追溯到根节点的完整上级链条。

**Consequences (testable):**
- 展示每一级上级的昵称、角色
- 根节点用户显示"无上级"
- 链条以列表形式展示，按层级排序

---

### 4.4 账号销售

**Description:** 用户（代理/经销商）可销售足球舆情系统账号给终端客户。销售时消耗自身账号额度，系统自动计算各级佣金。

**Functional Requirements:**

#### FR-11: 账号销售记录
代理/经销商在"销售账号"页面录入销售信息（客户手机号、备注），客户手机号需通过短信验证码验证。提交后消耗 1 个账号额度，系统自动记账各级佣金。

**Consequences (testable):**
- 销售时校验账号额度 > 0
- 客户手机号需短信验证通过后方可提交
- 销售成功后额度减 1
- 系统自动记账：自销佣金 + 各级上级的账号销售提成
- 销售记录包含：销售人、客户手机号、时间、金额 888 元
- 所有佣金以"记账"形式记录，不涉及真实资金流转

#### FR-12: 普通用户推荐购买
普通用户可推荐新用户购买账号，获得推荐返佣。`[ASSUMPTION: 普通用户的推荐通过其邀请码注册 + 被推荐人购买账号触发]`

**Consequences (testable):**
- 被推荐人通过普通用户的邀请码注册并购买账号后，普通用户获得 177.6 元返佣
- 返佣记录在"我的收益"中可见

---

### 4.5 佣金计算与收益

**Description:** 系统根据业务规则自动记账各级用户的佣金。佣金分为首次奖励、账号销售提成、长期奖励三类。所有佣金以"记账"形式计入用户余额，不涉及真实资金流转。所有佣金比例、金额参数均可在管理员后台配置。

**Functional Requirements:**

#### FR-13: 佣金规则配置（管理员可配）
系统按以下默认规则自动记账佣金（单位：元），所有数值参数均可在管理员后台修改：

| 行为场景 | 代理收益 | 经销商收益 | 普通用户收益 |
|----------|----------|------------|-------------|
| 自销 1 个账号 | 488.4 | 355.2 | — |
| 推荐新用户购买 | — | — | 177.6 |
| 发展 1 个经销商 | 2750 | 2000 | — |
| 推荐 1 个代理 | 5500 | 4000 | — |
| 下级销售 1 个账号 | 133.2 | — | — |
| 被推荐团队长期业绩提成 | 5% | 4% | — |

**Consequences (testable):**
- 每笔佣金记账精确到小数点后 2 位
- 佣金记账日志可追溯（哪笔业务触发、计算过程、归属人）
- 管理员可在后台修改所有佣金比例和金额参数（准入费、账号定价、各角色各场景的佣金比例/金额）
- 参数修改后仅对新产生的业务生效，不影响历史记录

#### FR-14: 我的收益
用户可在"我的收益"页面查看收益明细。页面需明确区分"记账余额"（系统记录的佣金，需提现才能到账）和"已提现金额"（已完成打款的金额），避免用户将记账余额误认为可随时使用的现金。

**Consequences (testable):**
- 页面顶部展示：记账余额（待提现）、已提现总额、可用余额（= 记账余额 - 已冻结提现金额）
- "记账余额"使用独立的视觉样式（如账本图标 + 说明文字"此为记账金额，需提交工单提现"）
- 收益明细按时间倒序排列
- 每条明细包含：金额、类型（首次奖励/账号提成/长期奖励/推荐返佣）、来源（下级昵称或账号销售）、时间
- 支持按类型筛选

#### FR-15: 长期奖励记账
当被推荐代理/经销商的团队产生销售业绩时，系统按比例记账上级的长期奖励。"团队业绩"定义为该用户及其全部下级（含间接下级）的账号销售总额。长期奖励沿邀请链逐级向上追溯，每一级按其直接上级的角色计算对应比例。结算周期由管理员在后台配置（默认按月，每月 1 日自动计算上月业绩）。

**Consequences (testable):**
- 代理的上级（代理）获得下级代理团队业绩的 5%（团队 = 该代理及其全部下级）
- 代理的上级（经销商）获得下级代理团队业绩的 4%
- 经销商的上级（代理）获得下级经销商团队业绩的 5%
- 经销商的上级（经销商）获得下级经销商团队业绩的 4%
- 长期奖励沿邀请链逐级向上追溯，每级独立计算
- 经销商发展代理后，即使代理角色高于经销商，佣金关系持续有效
- 结算周期可由管理员在后台配置（按月/按周/按天）
- 长期奖励以"记账"形式计入余额

---

### 4.6 提现工单

**Description:** 用户可提交提现申请，管理员在后台审核并线下打款后结单。v1 不集成在线支付提现。

**Functional Requirements:**

#### FR-16: 提现申请
用户在"我的收益"页面发起提现申请，填写提现金额和收款信息（银行卡/支付宝等）。提交后系统展示工单状态和预计处理时效。

**Consequences (testable):**
- 提现金额不能超过可用余额
- 提现金额需大于最低提现额度（默认 100 元，管理员可在后台配置）
- 提交后生成工单，状态为"待处理"，展示预计处理时效（如"通常 3 个工作日内处理"）
- 可用余额相应扣减（冻结）
- 用户可查看工单列表，每个工单展示：状态（待处理/已打款/已拒绝）、提交时间、金额
- 工单状态变更时通过订阅消息通知用户
- 管理员拒绝工单时需填写原因，用户端可见拒绝原因

#### FR-17: 工单管理（管理员）
管理员在后台查看所有提现工单，审核后标记"已打款"。

**Consequences (testable):**
- 工单列表支持按状态筛选（待处理/已打款/已拒绝）
- 管理员可查看工单详情（申请人、金额、收款信息、佣金来源）
- 管理员确认打款后工单状态变更为"已打款"
- 管理员可拒绝工单并填写原因，拒绝后金额退回用户余额
- 工单状态变更后用户端同步更新

### 4.7 管理员后台

**Description:** 独立的后台管理系统（Web 端），供管理员使用。

**Functional Requirements:**

#### FR-18: 用户管理
管理员可查看所有注册用户列表，支持搜索（手机号/昵称/邀请码），查看用户详情（角色、团队、收益）。

**Consequences (testable):**
- 用户列表分页展示
- 支持按角色筛选
- 用户详情包含：基本信息、上级信息、团队统计、收益汇总

#### FR-19: 准入审核
管理员查看待审核的准入申请，确认收款后批准。

**Consequences (testable):**
- 待审核列表按申请时间排序
- 批准后自动触发角色变更、额度分配、上级奖励计算
- 支持拒绝并填写原因

#### FR-20: 运营数据看板
管理员可在首页查看核心运营数据。

**Consequences (testable):**
- 展示：总用户数、代理数、经销商数、普通用户数
- 展示：今日新增用户、今日销售额
- 展示：待处理工单数
- `[ASSUMPTION: 不做复杂图表，仅数字卡片 + 简单列表]`

#### FR-21: 系统参数配置
管理员可在后台配置所有佣金和业务参数。

**Consequences (testable):**
- 可配置项包括：准入费金额（代理/经销商）、账号定价、账号配额（代理/经销商）、各角色各场景的佣金比例/金额、长期奖励结算周期、最低提现金额
- 参数修改后仅对新产生的业务生效，不影响历史记录
- 配置变更记录操作日志（谁、何时、改了什么、旧值、新值）

### 4.9 跨领域非功能需求

#### NFR-1: 佣金记账幂等性
每笔佣金记账操作必须基于唯一业务 ID（如销售记录 ID）进行幂等保护，防止重复记账。

**Consequences (testable):**
- 同一笔销售记录不会产生重复的佣金记账
- 系统在记账前检查业务 ID 是否已处理
- 幂等性保护覆盖：首次奖励、账号销售提成、长期奖励

#### NFR-2: 审计日志
所有涉及金额变动的操作（佣金记账、提现申请、工单状态变更、管理员配置修改）必须记录完整的审计日志。

**Consequences (testable):**
- 审计日志包含：操作时间、操作人、操作类型、变更前后值、关联业务 ID
- 审计日志不可删除或修改
- 管理员可在后台查看审计日志

#### NFR-3: 邀请码安全性
邀请码必须包含防篡改签名，确保角色信息不可伪造。

**Consequences (testable):**
- 邀请码使用 HMAC 或类似签名机制
- 签名密钥仅服务端持有
- 被篡改的邀请码解析失败

用户在以下场景收到微信小程序订阅消息通知：

## 5. Non-Goals (Explicit)

- **不做在线支付**：准入费和账号购买均为线下交易，系统仅记录和管理
- **不做在线提现**：提现通过工单 + 线下打款方式
- **不做复杂 BI 图表**：数据看板仅提供核心数字统计
- **不做多端适配**：v1 仅微信小程序 + Web 管理后台
- **不做角色升级**：v1 不支持经销商升级为代理（需重新注册）
- **不做多语言**
- **不做社交功能**（即时通讯、动态、评论等）

---

## 6. MVP Scope

### 6.1 In Scope

- 微信授权登录 + 手机号绑定
- 邀请码生成、填写、关系链记录
- 三级角色体系（普通用户/经销商/代理）
- 准入申请 + 管理员审核
- 账号额度管理与销售记录
- 佣金自动计算（首次奖励 + 账号提成 + 长期奖励）
- 我的团队（下级树）+ 我的上级链条
- 我的收益（明细列表）
- 提现工单（申请 + 审核 + 打款确认）
- 管理员后台（用户管理 + 准入审核 + 数据看板）
- 小程序订阅消息通知

### 6.2 Out of Scope for MVP

- 微信支付集成 → v2
- 在线提现（微信零钱/银行卡打款） → v2
- 经销商升级为代理 → v2
- 复杂数据图表（折线图、饼图等） → v2
- H5 / 支付宝小程序 / 其他端 → v2
- 多语言支持 → v2
- 角色权限的细粒度 RBAC → v2

## 7. Success Metrics

**Primary**
- **SM-1**: 注册转化率 — 通过邀请码进入的用户完成注册的比例 > 80%。Validates FR-1, FR-2, FR-3.
- **SM-2**: 佣金计算准确率 — 所有自动计算的佣金与人工核算一致，零误差。Validates FR-13, FR-14, FR-15.

**Secondary**
- **SM-3**: 工单处理时效 — 从提现申请到管理员打款确认的平均时长 < 48 小时。Validates FR-16, FR-17.
- **SM-4**: 系统可用性 — 小程序和后台管理系统可用率 > 99.5%。

**Counter-metrics (do not optimize)**
- **SM-C1**: 注册速度 — 不应为了快速注册而简化邀请码验证或手机号校验，防止关系链错误。
