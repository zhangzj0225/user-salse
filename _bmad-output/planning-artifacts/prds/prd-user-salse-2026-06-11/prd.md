# PRD: 用户分销系统

## 0. Document Purpose

本文档面向开发团队，定义「用户分销系统」的完整功能需求。本系统与足球舆情分析系统（业务系统）配合使用，边界明确：**分销系统负责代理/经销商管理、佣金计算、License 生成与验证、在线支付**，业务系统负责舆情功能展示和终端用户服务。两系统通过 License Code 对接，用户体系完全独立。

上游输入：`需求说明.md`、`用户旅途说明.md`（业务规则文档）、`sprint-change-proposal-2026-06-21.md`（对接需求变更）。

**v2 变更摘要**（2026-06-21）：取消开放注册 → 仅代理/经销商/管理员；邀请码 → 推荐码（持久码）；佣金 3 档 → 2 档；License 不绑定邮箱；新增在线支付小闭环；超管可创建种子用户。

---

## 1. Vision

本系统是一个**用户分销管理工具**，核心解决三个问题：

1. **分销裂变**：代理和经销商通过推荐码发展下级，支付时标识推荐关系，系统自动建立佣金归属链。首批代理/经销商由超管创建或通过在线支付自主开通。
2. **佣金自动计算**：根据角色和支付行为，系统自动计算各级佣金（首次奖励 + 后续收益 + 长期奖励）。
3. **License 对接**：代理/经销商通过额度销售或在线支付生成 License，业务系统通过验证 API 激活高级功能。两系统用户独立，License Code 做唯一桥梁。

系统定位为**与业务系统解耦的分销引擎**——分销逻辑稳定不变，业务系统可独立迭代热点应用。

## 2. Target User

### 2.1 角色定义

| 角色 | 所属系统 | 准入方式 | 可售额度 | License | 推荐码 |
|------|---------|---------|---------|---------|--------|
| **管理员** | 分销系统 | 命令行创建 | — | — | ✗ |
| **代理** | 分销系统 | 超管创建 或 在线支付 10000 | 22 | ✓ | ✓ |
| **经销商** | 分销系统 | 超管创建 或 在线支付 5000 | 11 | ✓ | ✓ |
| **终端用户** | 业务系统 | 业务系统开放注册 | — | 使用方 | ✗ |

**关键变化**：终端用户（原"普通用户/888会员"）不在分销系统注册。分销系统仅含代理、经销商和管理员。888 会员概念移至业务系统侧，通过 License 激活。

### 2.2 Jobs To Be Done

- 作为代理，我需要通过推荐码发展下级，下级支付时我获得佣金
- 作为经销商，我需要通过推荐码发展下级，下级支付时我获得佣金
- 作为代理/经销商，我需要用额度生成 License 分发给客户
- 作为超管，我需要创建种子代理/经销商、处理提现工单、查看运营数据
- 作为终端用户，我需要通过支付 888 获得 License，在业务系统激活高级功能

### 2.3 Non-Users (v1)

- 终端用户不直接使用分销系统（仅使用支付页面和业务系统）
- 不面向非 H5 浏览器环境（v1 仅 H5 + Web 管理后台）

### 2.4 Key User Journeys

- **UJ-1. 超管创建种子代理。**
  - **Persona + context:** 超管小王需要初始化分销网络。
  - **Entry state:** 已登录管理后台。
  - **Path:** 进入"用户管理" → 点击"创建用户" → 填写邮箱 + 选择角色"代理" + 可选填推荐码（上级） → 系统创建代理账号 + 分配 22 额度 + 生成 License + 生成推荐码 → 代理收到邮件通知。
  - **Climax:** 种子代理创建完成，可用邮箱验证码登录分销系统。
  - **Resolution:** 代理拥有推荐码，可开始发展下级。

- **UJ-2. 代理用额度生成 License 卖给客户。**
  - **Persona + context:** 李代理有客户想购买高级功能。
  - **Entry state:** 李代理已登录分销系统，有剩余额度。
  - **Path:** 李代理进入"额度管理" → 点击"生成 License" → 消耗 1 个额度 → 获得未激活的 License Code → 李代理把 Code 交付给客户。
  - **Climax:** 客户在业务系统输入 Code → 业务系统调分销系统验证 API → 验证通过 → License 标记已激活 + 绑定业务系统用户标识。
  - **Resolution:** 李代理额度减 1，客户在业务系统获得高级功能。不产生佣金。

- **UJ-3. 终端用户通过支付购买 License。**
  - **Persona + context:** 用户在业务系统看到"升级"按钮。
  - **Entry state:** 用户已在业务系统注册登录。
  - **Path:** 点击"升级" → 跳转到分销系统支付页 `/pay?amount=888&redirect=...` → 填写邮箱 + 可选填推荐码 → 扫码支付 888 → 支付成功 → 生成 License → 页面展示 + 邮件发送 → 点击"返回业务系统" → 自动填入 License → 激活高级功能。
  - **Climax:** 如填了推荐码，推荐人（代理/经销商）获得首次奖励佣金。
  - **Resolution:** 用户获得高级功能，推荐人获得佣金记账。

- **UJ-4. 代理查看团队和收益。**
  - **Persona + context:** 赵代理想了解团队结构和佣金收益。
  - **Entry state:** 已登录分销系统。
  - **Path:** 进入"我的团队" → 查看直接下级列表（仅代理/经销商）→ 进入"我的收益" → 查看记账余额和明细。
  - **Climax:** 清晰看到团队规模和各项佣金入账。
  - **Resolution:** 赵代理了解团队全貌和收益情况。

- **UJ-5. 管理员处理提现工单。**
  - **Persona + context:** 管理员小王收到代理的提现申请。
  - **Entry state:** 已登录管理后台。
  - **Path:** 进入"工单管理" → 查看待处理提现工单 → 查看详情 → 确认无误后点击"已打款" → 系统更新工单状态。
  - **Climax:** 工单状态变更，申请人收到通知。
  - **Resolution:** 提现流程闭环，财务记录可追溯。

## 3. Glossary

- **分销系统用户** — 代理、经销商或管理员，通过邮箱验证码登录分销系统。
- **终端用户** — 在业务系统注册的用户，不在分销系统注册。通过 License 获得高级功能。
- **推荐码** — 代理/经销商的身份标识码，持久有效，可重复使用。用于支付时标识推荐人，建立佣金归属关系。包含 HMAC-SHA256 防篡改签名。
- **上级/下级** — 通过推荐码建立的直接推荐关系。A 的推荐码被 B 在支付时填写，则 A 是 B 的直接上级，B 是 A 的直接下级。
- **团队** — 某用户的所有下级（含间接下级）构成的树形结构。仅含代理/经销商。
- **角色** — 分销系统用户身份：代理、经销商。角色由支付金额决定（5000→经销商，10000→代理）。
- **可售额度** — 代理（22 个）或经销商（11 个）可用来生成 License 的数量。
- **License** — 授权码，用于在业务系统中激活高级功能。生成时不绑定用户，激活时绑定业务系统用户标识。两种获取方式：额度销售（场景 A）、在线支付（场景 B）。
- **场景 A（额度销售）** — 代理/经销商消耗自身可售额度生成 License，交付给客户。**不产生佣金**。
- **场景 B（在线支付）** — 终端用户或新代理/经销商通过支付小闭环在线支付 888/5000/10000。**填了推荐码则产生佣金**。
- **首次奖励** — 场景 B 中，支付时填了推荐码，推荐人获得的一次性佣金。
- **后续收益** — 仅代理→经销商关系：经销商的推荐码用户支付 888，经销商的上级代理获得 133.2 元/笔。
- **长期奖励** — 定期结算，基于直接下级的全部收入总和 × 比例。
- **记账** — 系统对佣金的计量记录，不代表真实资金已到账。用户需通过提现工单转为实际收款。
- **工单** — 用户发起的提现申请记录，由管理员审核并线下打款后结单。
- **支付小闭环** — 分销系统提供的独立支付页面，支持 888/5000/10000 在线支付。

## 4. Features

### 4.1 用户认证

**Description:** 分销系统仅面向代理、经销商和管理员。`/login` 使用邮箱验证码登录已注册用户，不创建新用户。代理/经销商的创建通过两条路径：(1) 超管在管理后台直接创建；(2) 通过支付小闭环在线支付 5000/10000 后自动创建账号。

**Functional Requirements:**

#### FR-1: 邮箱验证码登录
代理/经销商/管理员通过邮箱接收验证码登录分销系统。不提供注册页面，不创建新用户。

**Consequences (testable):**
- 未登录用户无法进入任何功能页面
- 验证码 5 分钟有效，过期需重新获取
- `/login`：邮箱已存在则发送验证码，不存在则提示"用户不存在"
- 登录成功后返回 JWT（type=user）
- 不提供 `/register` 页面

#### FR-2: 邮箱即分销系统账号标识
邮箱是代理/经销商/管理员在分销系统的唯一账号标识。终端用户在业务系统有独立的账号体系，不在分销系统注册。两系统通过 License Code 对接，不共享用户体系。

**Consequences (testable):**
- 邮箱全局唯一
- 邮箱格式不合法时提示错误
- 终端用户不在分销系统有账号

#### FR-3: 推荐码
推荐码是代理/经销商的身份标识码，用于支付时标识推荐人。推荐码为持久码，每人 1 个，可重复使用。用户在支付小闭环支付 888/5000/10000 时可选填推荐码。填了推荐码 → 计算佣金；不填 → 无佣金，平台收全款。

**Consequences (testable):**
- 推荐码持久有效，不消耗
- 每个代理/经销商 1 个固定推荐码
- 推荐码包含 HMAC-SHA256 防篡改签名
- 支付时选填，不强制
- 推荐码不建立"注册关系"，只建立"佣金归属关系"
- 被篡改的推荐码解析失败，提示"推荐码无效"
- 用户不能填写自己的推荐码

#### FR-4: 推荐码生成
代理/经销商创建时（超管创建或支付成功），系统自动生成 1 个持久推荐码。管理员不生成推荐码。推荐码在用户生命周期内不变。

**Consequences (testable):**
- 代理/经销商创建时自动生成推荐码
- 推荐码全局唯一，包含防篡改签名（HMAC-SHA256）
- 推荐码持久有效，不消耗，不可重置
- 用户可在"我的"页面查看和复制自己的推荐码

### 4.2 角色与支付

**Description:** 代理/经销商角色由支付金额决定。支持两条路径：(1) 超管直接创建种子用户；(2) 通过支付小闭环在线支付。各充值独立，不互斥，不升级。

**Functional Requirements:**

#### FR-5: 在线支付（场景 B 入口）
用户通过支付小闭环页面在线支付 888/5000/10000。支付成功后自动触发对应权益。

**Consequences (testable):**
- 支付页面 URL：`/pay?amount={888|5000|10000}&referral={推荐码}&redirect={回跳URL}`
- 支付方式：微信扫码 / 支付宝扫码
- 必填信息：邮箱
- 选填信息：推荐码
- 支 888 → 生成 License（终端用户购买会员）
- 支 5000 → 创建经销商账号 + 11 额度 + License + 推荐码
- 支 10000 → 创建代理账号 + 22 额度 + License + 推荐码
- 5000/10000 支付后，邮箱成为分销系统登录账号
- 各充值独立，经销商可再付 10000 成为代理（不退 5000，额度累加）
- 支付回调验签 + 幂等保护

#### FR-6: 线下支付审核（管理员后台）
管理员可手动确认线下收款，为用户开通角色。保留此路径用于线下交易场景。

**Consequences (testable):**
- 管理员确认后用户角色变更为对应角色
- 管理员确认后系统自动分配可售额度（代理 22 个，经销商 11 个）
- 管理员确认后系统自动生成 License
- 管理员确认后系统自动记账推荐人的首次奖励（如有）
- 管理员可拒绝并填写原因
- 所有佣金以"记账"形式记录

#### FR-7: 可售额度管理
代理/经销商可在"我的额度"中查看剩余可售额度、已生成 License 列表。额度耗尽后可向管理员申请补购。

**Consequences (testable):**
- 代理初始额度 22 个，经销商 11 个
- 生成 License 后额度减 1
- 额度为 0 时不可继续生成 License
- 额度耗尽后显示"申请补购"入口
- 管理员审核补购申请，确认后追加额度

---

### 4.3 推荐链与团队管理

**Description:** 系统记录完整的推荐关系链，每个用户可查看上级链条和下级团队树。团队仅含代理/经销商。

**Functional Requirements:**

#### FR-8: 推荐关系记录
系统在用户支付时填写推荐码后，根据支付金额区分关系建立行为。

**Consequences (testable):**
- 关系数据包含：上级用户 ID、下级用户 ID（仅 5000/10000）、推荐码、支付记录 ID、创建时间
- 5000/10000 支付 + 推荐码 → 建立团队关系（上下级均为分销系统用户）
- 888 支付 + 推荐码 → 仅记录佣金归属，不建立团队关系（支付者为终端用户，不在分销系统）
- 无推荐码的支付不建立推荐关系，不产生佣金
- 关系不可删除或修改
- 支持通过关系表追溯完整链条

#### FR-9: 我的团队（下级树）
用户可查看以自己为根节点的下级团队树。

**Consequences (testable):**
- 展示每个直接下级的邮箱、角色、注册时间、直接下级数量
- 支持逐层展开，不限层级深度
- 展示团队总人数统计
- 团队仅含代理/经销商

#### FR-10: 我的上级链条
用户可查看从自己向上追溯到根节点的完整上级链条。

**Consequences (testable):**
- 展示每一级上级的邮箱、角色
- 根节点用户显示"无上级"
- 链条按层级排序

---

### 4.4 License 生成与额度销售

**Description:** 两个场景：(A) 代理/经销商用额度生成 License（不产生佣金），(B) 终端用户在线支付 888 获得 License（填推荐码产生佣金）。

**Functional Requirements:**

#### FR-11: 额度销售生成 License（场景 A）
代理/经销商在后台点击"生成 License"，消耗 1 个可售额度，生成一个未激活的 License Code。License 生成时不绑定任何用户信息。代理将 License Code 交付给客户，客户在业务系统激活。

**Consequences (testable):**
- 生成时校验可售额度 > 0
- 生成后额度减 1
- License 状态为 `unused`（未激活）
- License 不绑定邮箱或用户
- 不产生任何佣金记账
- 生成记录包含：生成人、时间、License Code

#### FR-12: 在线支付生成 License（场景 B）
终端用户通过支付小闭环支付 888，系统生成 License。如填了推荐码则计算佣金。

**Consequences (testable):**
- 支付 888：生成 License，发至用户邮箱
- 填了推荐码（代理）：推荐人获 488.4 元（55%）
- 填了推荐码（经销商）：推荐人获 355.2 元（40%）
- 经销商的推荐码用户支付 888：经销商的上级代理获 133.2 元（后续收益）
- 无推荐码：无佣金，平台收全款
- 所有佣金以"记账"形式记录

---

### 4.5 佣金计算与收益

**Description:** 系统根据角色和支付行为自动记账佣金。佣金分为首次奖励、后续收益、长期奖励三类。所有佣金以"记账"形式计入余额。所有参数均可在管理员后台配置。

**Functional Requirements:**

#### FR-13: 佣金规则配置（管理员可配）

**首次奖励（场景 B，支付时填了推荐码，推荐人获得）：**

| 支付金额 | 推荐人=代理 (55%) | 推荐人=经销商 (40%) | 无推荐码 |
|---------|:---:|:---:|:---:|
| 888 元 | 488.4 元 | 355.2 元 | — |
| 5000 元 | 2750 元 | 2000 元 | — |
| 10000 元 | 5500 元 | 4000 元 | — |

**后续收益（仅代理→经销商链）：**
- 经销商的推荐码用户支付 888，经销商的上级代理获得 133.2 元/笔

**长期奖励（定期结算，仅限直接下级的全部收入总和 × 比例）：**

| 上级角色 | 下级角色 | 长期奖励比例 |
|---------|---------|------------|
| 代理 | 代理 | 5% |
| 代理 | 经销商 | 不适用（已由 133.2 元/笔替代） |
| 经销商 | 代理 | 4% |
| 经销商 | 经销商 | 4% |

**"全部收入"定义：** 直接下级获得的首次奖励 + 后续收益 + 长期奖励总和。额度销售不产生收入，不计入。

**Consequences (testable):**
- 每笔佣金记账精确到小数点后 2 位
- 佣金记账日志可追溯
- 管理员可在后台修改所有佣金比例和金额参数
- 参数修改后仅对新产生的业务生效
- 无推荐码的支付不产生任何佣金

#### FR-14: 我的收益
用户可在"我的收益"页面查看收益明细。

**Consequences (testable):**
- 页面顶部展示：记账余额（待提现）、已提现总额、可用余额（= 记账余额 - 已冻结提现金额）
- "记账余额"使用独立的视觉样式（账本图标 + 说明文字"此为记账金额，需提交工单提现"）
- 收益明细按时间倒序排列
- 每条明细包含：金额、类型（首次奖励/后续收益/长期奖励）、来源、时间
- 支持按类型筛选

#### FR-15: 长期奖励记账
系统按周期（默认按月）自动计算长期奖励。基于直接下级的全部收入总和 × 对应比例。

**Consequences (testable):**
- 仅计算直接下级的收入，不穿透更深层级
- 代理→经销商关系不适用长期奖励（已由 133.2 元/笔替代）
- 代理→代理关系：5%
- 经销商→代理关系：4%
- 经销商→经销商关系：4%
- 结算周期可由管理员配置（按月/按周/按天）
- 长期奖励以"记账"形式计入余额
- 幂等保护：同一周期不重复结算

---

### 4.6 License 对接

**Description:** 代理/经销商通过额度销售或在线支付生成 License Code。业务系统通过调用本系统的验证 API 确认 License 有效性并激活。

**Functional Requirements:**

#### FR-16: License 生成
License 在额度销售或在线支付 888 后自动生成。生成时不绑定用户信息，激活时绑定业务系统用户标识。

**Consequences (testable):**
- License 在额度销售或支付成功后自动生成
- License 生成时不绑定邮箱
- License 两种状态：`unused`（未激活）→ `active`（已激活）
- License 一次性使用，激活后不可再用
- License 格式为唯一字符串（含 HMAC-SHA256 防篡改签名）
- 代理/经销商可在"我的额度"页面查看已生成的 License 列表

#### FR-17: License 验证接口
提供 API 供业务系统验证 License 有效性。

**Consequences (testable):**
- 接口接收：License Code + 业务系统用户标识（可选）
- 验证逻辑：License 存在 + 未激活
- 验证通过：标记为已激活（`active`），绑定业务系统用户标识（`activated_user_ref`），返回成功
- 已激活的 License 再次验证返回"已激活"状态
- 验证失败：返回具体错误（License 不存在 / 已激活）
- 接口需 API Key 鉴权

---

### 4.7 提现工单

**Description:** 代理/经销商可提交提现申请，管理员审核并线下打款。

**Functional Requirements:**

#### FR-18: 提现申请
用户在"我的收益"页面发起提现申请。

**Consequences (testable):**
- 提现金额不能超过可用余额
- 提现金额需大于最低提现额度（默认 100 元，可配置）
- 提交后生成工单，状态为"待处理"
- 可用余额相应冻结
- 用户可查看工单列表
- 工单状态变更时通知用户

#### FR-19: 工单管理（管理员）
管理员审核提现工单。

**Consequences (testable):**
- 工单列表支持按状态筛选
- 管理员确认打款后状态变更为"已打款"
- 管理员可拒绝并填写原因，金额退回余额
- 工单状态变更后用户端同步更新

---

### 4.8 管理员后台

**Description:** 独立的后台管理系统（Web 端），供超管管理基础数据和工单。

**Functional Requirements:**

#### FR-20: 用户管理 + 创建种子用户
管理员可查看所有用户列表，并可直接创建种子用户（代理/经销商）。

**Consequences (testable):**
- 用户列表分页展示（默认 page_size=20）
- 支持按邮箱搜索
- 支持按角色筛选（代理/经销商）
- 用户详情包含：基本信息、上级信息、团队统计、收益汇总
- **创建种子用户**：指定邮箱 + 角色（代理/经销商）+ 可选填推荐码（上级）→ 系统直接创建
- 跳过支付流程，自动分配额度（代理 22 / 经销商 11）
- 自动生成个人 License + 推荐码
- 种子用户用邮箱验证码登录分销系统

#### FR-21: 支付审核
管理员查看待审核的支付记录（线下支付路径），确认收款后批准。

**Consequences (testable):**
- 待审核列表按申请时间排序
- 批准后自动触发角色变更、额度分配、License 生成、佣金记账（如有推荐码）
- 支持拒绝并填写原因

#### FR-22: 运营数据看板
管理员可在首页查看核心运营数据。

**Consequences (testable):**
- 展示：总用户数、代理数、经销商数
- 展示：今日新增用户、今日支付总额（含在线支付）
- 展示：待处理工单数
- 展示：今日 License 生成数、激活数
- 仅数字卡片，不做复杂图表

#### FR-23: 系统参数配置
管理员可在后台配置所有参数。

**Consequences (testable):**
- 可配置项：支付金额（888/5000/10000）、可售额度（代理/经销商）、各角色佣金比例、长期奖励结算周期、最低提现金额
- 参数修改仅对新业务生效
- 配置变更记录操作日志

### 4.9 消息通知

**Description:** 系统在关键事件发生时向用户推送通知。

**Functional Requirements:**

#### FR-24: 站内/邮件消息通知
用户在以下场景收到通知（H5 站内消息 + 邮件）：
- 下级支付（有人通过你的推荐码支付）
- 佣金入账
- 工单状态变更（已打款/已拒绝）
- 支付审核通过
- 种子用户创建通知（含登录引导）

**Consequences (testable):**
- 通知内容包含事件类型、金额（如适用）、时间

### 4.10 在线支付小闭环

**Description:** 分销系统提供独立的支付页面，支持 888/5000/10000 三档在线支付。支付成功后自动生成 License（888）或创建账号+分配额度+生成License（5000/10000），并计算佣金。

**Functional Requirements:**

#### FR-25: 支付小闭环
分销系统提供独立部署的支付页面，对接微信/支付宝扫码支付。

**Consequences (testable):**
- 支付页面 URL：`/pay?amount={888|5000|10000}&referral={推荐码}&redirect={回跳URL}`
- 必填：邮箱（用于接收 License / 创建账号）
- 选填：推荐码（用于计算佣金）
- 支付方式：微信扫码 / 支付宝扫码
- 888 支付：生成 License → 邮件发送 + 页面展示 → 引导回业务系统激活
- 5000 支付：创建经销商账号 + 11 额度 + License + 推荐码 → 计算佣金
- 10000 支付：创建代理账号 + 22 额度 + License + 推荐码 → 计算佣金
- 支付回调验签 + 幂等保护
- `redirect` 参数白名单校验（防开放重定向）
- 支付成功后 License Code 通过 URL 参数回传给业务系统
- 支付失败提供重试入口

---

### 4.11 系统对接说明

**Description:** 分销系统与业务系统（舆情系统）通过 License Code 对接，用户体系完全独立。

**对接边界：**

| 职责 | 用户分销系统（本项目） | 业务系统（舆情系统） |
|------|---------------------|----------------|
| 终端用户注册 | 不涉及 | **主**（开放注册） |
| 代理/经销商管理 | **主** | 不涉及 |
| 用户登录 | 各自独立（邮箱验证码） | 各自独立 |
| 推荐码 | **主** | 不涉及 |
| 角色管理 | **主**（代理/经销商） | 不涉及（终端用户角色自管） |
| License 生成 | **主** | 不涉及 |
| License 验证 | **提供 API** | 调用 API |
| 在线支付 | **主**（支付小闭环） | 跳转入口 |
| 佣金/提现 | **主** | 不涉及 |
| 舆情数据 | 不涉及 | **主** |

**对接流程：**

路径 A — 额度销售：
1. 代理在分销系统生成 License（消耗额度）
2. 代理把 License 交付给客户
3. 客户在业务系统输入 License
4. 业务系统调分销系统 `POST /api/v1/license/verify` 验证
5. 验证通过 → License 绑定业务系统用户标识 → 业务系统开通高级功能

路径 B — 在线支付：
1. 用户在业务系统点击"升级"
2. 跳转到分销系统支付页 `/pay?amount=888&redirect=...`
3. 用户填写邮箱 + 可选填推荐码 → 扫码支付
4. 支付成功 → 分销系统生成 License → 计算佣金（如有推荐码）
5. License 通过页面展示 + 邮件发送
6. 用户点击"返回业务系统" → License Code 通过 URL 参数回传
7. 业务系统自动填入 License → 调验证 API → 开通高级功能

### 4.12 跨领域非功能需求

#### NFR-1: 佣金记账幂等性
每笔佣金记账基于唯一业务 ID 进行幂等保护。

**Consequences (testable):**
- 同一笔业务不会产生重复佣金记账
- 幂等保护覆盖：首次奖励、后续收益、长期奖励

#### NFR-2: 审计日志
所有金额变动操作记录完整审计日志。

**Consequences (testable):**
- 审计日志包含：操作时间、操作人、操作类型、变更前后值、关联业务 ID
- 审计日志不可删除或修改

#### NFR-3: 推荐码安全性
推荐码包含防篡改签名。

**Consequences (testable):**
- 推荐码使用 HMAC-SHA256 签名
- 签名密钥仅服务端持有
- 被篡改的推荐码解析失败

#### NFR-4: License 安全性
License Code 防伪造、防转让。

**Consequences (testable):**
- License 一次性使用，激活后不可再用
- License 验证接口需 API Key 鉴权
- License 激活时绑定业务系统用户标识，防多设备共用

#### NFR-5: 支付安全
在线支付需防止伪造支付通知和重复处理。

**Consequences (testable):**
- 支付回调验签（微信/支付宝官方签名）
- 支付回调幂等处理（同一笔支付不重复触发权益）
- `redirect` 参数白名单校验

## 5. Non-Goals (Explicit)

- **不做在线提现**：提现通过工单 + 线下打款
- **不做支付退款**：各支付独立，支付成功后不支持退款
- **不做复杂 BI 图表**：仅数字卡片
- **不做多端适配**：v1 仅 H5 + Web 管理后台
- **不做 SSO**：两系统各自独立登录，License Code 做桥梁
- **不做退款**：支付成功后不支持退款
- **不做社交功能**

## 6. MVP Scope

### 6.1 In Scope

- 邮箱验证码登录（仅已注册用户，不创建新用户）
- 推荐码生成（持久码，仅代理/经销商）
- 两级角色体系（经销商/代理），支付决定角色
- 超管创建种子用户（代理/经销商）
- 在线支付小闭环（888/5000/10000 扫码支付）
- 线下支付审核（管理员手动确认）
- 可售额度管理（额度销售场景 A）
- 推荐支付佣金计算（场景 B：首次奖励 2 档 + 后续收益 + 长期奖励）
- License 生成与验证接口（不绑定邮箱，激活时绑定业务系统用户）
- 我的团队（下级树）+ 我的上级链条
- 我的收益（明细列表）
- 提现工单（申请 + 审核 + 打款确认）
- 管理员后台（用户管理 + 创建种子 + 支付审核 + 数据看板 + 参数配置）
- H5 站内/邮件消息通知

### 6.2 Out of Scope for MVP

- 在线提现 → v2
- 复杂数据图表 → v2
- 原生 App → v2
- 多语言支持 → v2
- 退款机制 → v2

## 7. Success Metrics

**Primary**
- **SM-1**: 支付转化率 — 进入支付页面的用户完成支付的比例 > 70%
- **SM-2**: 佣金计算准确率 — 所有自动计算的佣金与人工核算一致，零误差

**Secondary**
- **SM-3**: 工单处理时效 — 从提现申请到管理员打款确认的平均时长 < 48 小时
- **SM-4**: License 验证成功率 — License 验证接口成功率 > 99.5%
- **SM-5**: 推荐码使用率 — 有推荐码的支付占总支付的比例 > 60%
