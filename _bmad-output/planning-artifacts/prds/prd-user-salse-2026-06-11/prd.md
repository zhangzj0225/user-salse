# PRD: 足球舆情系统 — 销售奖励系统

## 0. Document Purpose

本文档面向开发团队，定义「足球舆情系统」销售奖励系统的完整功能需求。本系统与足球舆情分析系统配合使用，边界明确：**本系统负责注册→邀请码→角色管理→888会员开通→生成License**，舆情系统负责 License 验证后的高级功能展示。

上游输入：`需求说明.md`、`用户旅途说明.md`（业务规则文档）。

---

## 1. Vision

本系统是一个基于 H5 的**销售奖励管理工具**，核心解决三个问题：

1. **邀请链传播**：用户通过邀请码注册建立上下级关系（分销裂变路径）；首批用户由管理员通过冷启动登录播种。充值金额决定角色（888会员/经销商/代理），邀请码只负责关系建立。
2. **佣金自动计算**：根据角色和充值行为，系统自动计算各级佣金（首次奖励 + 后续收益 + 长期奖励）。
3. **License 对接**：用户开通 888 会员后生成 License，凭 License 在舆情系统中激活高级功能。

系统定位为**轻量级对接工具**——通过邮箱关联两系统，License Code 做桥梁，两边可独立运行。

## 2. Target User

### 2.1 角色定义

| 角色 | 准入方式 | 可售额度 | License | 可生成邀请码 |
|------|---------|---------|---------|------------|
| **普通用户** | 免费注册 | 0 | ✗ | ✓ |
| **888 会员** | 充值 888 元 | 0 | ✓ | ✓ |
| **经销商** | 充值 5000 元 | 11 | ✓ | ✓ |
| **代理** | 充值 10000 元 | 22 | ✓ | ✓ |
| **管理员** | 种子数据创建 | — | — | — |

**角色由充值金额决定，不由邀请码决定。** 邀请码只建立上下级关系，不影响角色。

### 2.2 Jobs To Be Done

- 作为代理，我需要通过邀请码发展下线，下线充值时我获得佣金
- 作为经销商，我需要通过邀请码发展下线，下线充值时我获得佣金
- 作为普通用户，我需要推荐他人充 888，获得推荐返佣
- 作为管理员，我需要审核充值、处理提现工单、查看运营数据

### 2.3 Non-Users (v1)

- 不面向未注册的游客（所有功能需登录后使用）
- 不面向非 H5 浏览器环境（v1 仅 H5 + Web 管理后台）

### 2.4 Key User Journeys

- **UJ-1. 新用户通过邀请码注册并充值成为代理。**
  - **Persona + context:** 张先生收到朋友（代理）发来的邀请码。
  - **Entry state:** 未注册，通过朋友分享的 H5 链接进入。
  - **Path:** 打开 H5 → 邮箱验证码注册（填写邀请码，建立上下级关系，邀请码失效）→ 成为普通用户 → 线下支付 10000 元 → 管理员后台确认收款 → 角色变更为代理，获得 22 个可售额度 + License → 系统记账上级首次奖励 5500 元。
  - **Climax:** 管理员确认后，上下级关系可见，上级收益余额中新增 5500 元记账。
  - **Resolution:** 张先生成为代理，拥有自己的邀请码，可继续发展下线。

- **UJ-2. 代理用额度销售账号给客户。**
  - **Persona + context:** 李代理有客户想购买 888 会员。
  - **Entry state:** 李代理已登录，有剩余额度。
  - **Path:** 李代理进入"销售账号"页面 → 录入客户邮箱 → 邮箱验证 → 确认销售 → 消耗 1 个额度 → 客户成为 888 会员，生成 License → 客户的上级 = 李代理。
  - **Climax:** 不产生佣金（成本已在准入费中），客户获得 License 可去舆情系统使用。
  - **Resolution:** 李代理额度减 1，客户成为其下级 888 会员。

- **UJ-3. 用户查看自己的团队和收益。**
  - **Persona + context:** 赵代理想了解团队结构和佣金收益。
  - **Entry state:** 已登录。
  - **Path:** 进入"我的团队" → 查看直接下级列表 → 逐层展开 → 进入"我的收益" → 查看记账余额和明细。
  - **Climax:** 清晰看到团队规模和各项佣金入账。
  - **Resolution:** 赵代理了解团队全貌和收益情况。

- **UJ-4. 管理员处理提现工单。**
  - **Persona + context:** 管理员小王收到代理的提现申请。
  - **Entry state:** 已登录后台管理系统。
  - **Path:** 进入"工单管理" → 查看待处理提现工单 → 点击查看详情 → 确认无误后点击"已打款" → 系统更新工单状态。
  - **Climax:** 工单状态变更，申请人收到通知。
  - **Resolution:** 提现流程闭环，财务记录可追溯。

## 3. Glossary

- **用户** — 通过邮箱验证码完成注册/登录的自然人。每个用户拥有唯一 ID 和个人邀请码。
- **邀请码** — 用户生成的唯一标识字符串，用于建立上下级关系。**统一类型，不区分目标角色**。一次性使用，用后失效。包含防篡改签名。
- **上级/下级** — 通过邀请码建立的直接推荐关系。A 的邀请码被 B 注册时使用，则 A 是 B 的直接上级，B 是 A 的直接下级。
- **团队** — 某用户的所有下级（含间接下级）构成的树形结构。
- **角色** — 用户在分销体系中的身份：普通用户、888 会员、经销商、代理。**角色由充值金额决定**，不由邀请码决定。
- **充值** — 用户支付费用以获得更高角色或 License 的行为。充 888 → 888 会员，充 5000 → 经销商，充 10000 → 代理。各充值独立，不互斥，不升级。
- **可售额度** — 代理（22 个）或经销商（11 个）可用来销售给客户的 888 会员账号数量。充 10000/5000 时获得。
- **License** — 用户开通 888 会员（含代理/经销商自带）后生成的授权码，用于在舆情系统中激活高级功能。一次性使用，绑定邮箱。
- **场景 A（额度销售）** — 代理/经销商消耗自身可售额度，为客户开通 888 会员。**不产生佣金**，客户上级 = 销售者。
- **场景 B（推荐充值）** — 下级自己充值 888/5000/10000。**产生佣金**，按规则记账给直接上级。
- **首次奖励** — 场景 B 中，下级充值时直接上级获得的一次性佣金。
- **后续收益** — 仅代理→经销商关系：经销商每次推荐他人充 888（场景 B），代理获得 133.2 元/笔。
- **长期奖励** — 定期结算，基于直接下级的全部收入总和 × 比例。
- **记账** — 系统对佣金的计量记录，不代表真实资金已到账。用户需通过提现工单转为实际收款。
- **工单** — 用户发起的提现申请记录，由管理员审核并线下打款后结单。

## 4. Features

### 4.1 用户注册与认证

**Description:** 用户通过 H5 完成注册与登录，统一采用邮箱验证码机制。系统提供两条用户创建路径：

- **注册路径（`/register`，分销裂变）**：用户输入邮箱 → 获取注册验证码 → 填写邀请码 → 校验通过后建立上下级关系（`parent_id` = 邀请码生成者）→ 成为普通用户。
- **冷启动登录路径（`/login`，管理员播种）**：用于第一批根用户的创建与后续管理员/种子账号登录，无需邀请码，创建无 `parent_id` 的根节点用户。分销裂变形成后此路径主要服务于管理员登录。

注册成功后系统自动生成个人邀请码。

**Functional Requirements:**

#### FR-1: 邮箱验证码登录/注册
用户通过邮箱接收验证码完成身份验证。`/login` 与 `/register` 共用邮箱验证码机制，但使用不同 scene（`login` / `register`）以区分验证码用途。

**Consequences (testable):**
- 未登录用户无法进入任何功能页面
- 验证码 5 分钟有效，过期需重新获取
- `/login`：邮箱已存在则登录既有用户，不存在则冷启动创建根用户（无 `parent_id`）
- `/register`：邮箱已注册时提示"邮箱已注册"，不允许重复注册
- 登录/注册成功后返回 JWT（type=user）

#### FR-2: 邮箱即账号标识
邮箱是用户在本系统的唯一账号标识，也是两系统关联的桥梁。无需单独的绑定步骤——首次验证码通过即建立账号。

**Consequences (testable):**
- 邮箱全局唯一，注册时校验不重复
- 邮箱格式不合法时提示错误
- 邮箱验证通过后用户状态标记为"active"

#### FR-3: 邀请码填写与关系建立（注册路径）
用户在 `/register` 注册时**必须**填写邀请码。邀请码为统一类型，不区分目标角色。填写后建立上下级关系，邀请码失效。

**Consequences (testable):**
- 邀请码不存在时提示"邀请码无效"
- 邀请码已使用时提示"邀请码已被使用"
- 邀请码有效时建立上下级关系（`parent_id` = 邀请码生成者），邀请码标记为已使用（`used_by` + `used_at`）
- 注册路径邀请码必填（不填返回 422）
- 根节点用户不经注册路径产生，由 `/login` 冷启动创建（无 `parent_id`）
- 用户不能填写自己的邀请码
- 邀请码一次性使用，使用后不可再用

#### FR-4: 个人邀请码生成
注册成功后，系统为用户生成个人邀请码。所有角色均可生成邀请码，类型统一，不区分目标角色。邀请码包含防篡改签名。

**Consequences (testable):**
- 邀请码全局唯一，包含防篡改签名（HMAC-SHA256）
- 所有角色（普通用户/888会员/经销商/代理）均可生成邀请码
- 邀请码类型统一，不区分推荐目标
- 邀请码一次性使用，使用后失效
- 用户可在"我的"页面查看和复制自己的邀请码
- 可生成多个未使用的邀请码
- 被篡改的邀请码解析失败，提示"邀请码无效"

### 4.2 角色与充值

**Description:** 用户角色由充值金额决定。充值后角色变更，获得对应权益（可售额度 + License）。各充值独立，不互斥，不升级。

**Functional Requirements:**

#### FR-5: 充值申请
用户选择充值金额（888/5000/10000），系统生成待审核的充值记录。用户线下完成支付后，由管理员在后台确认。

**Consequences (testable):**
- 充 888 → 888 会员（获得 License，无可售额度）
- 充 5000 → 经销商（获得 License + 11 个可售额度）
- 充 10000 → 代理（获得 License + 22 个可售额度）
- 各充值独立，888 会员可再充 5000 成为经销商（不退 888）
- 提交后状态为"待审核"
- 管理员确认后角色变更，获得对应权益

#### FR-6: 充值确认（管理员后台）
管理员查看待审核的充值记录，确认收款后角色生效。系统自动记账上级的首次奖励。

**Consequences (testable):**
- 管理员确认后用户角色变更为对应角色
- 管理员确认后系统自动分配可售额度（代理 22 个，经销商 11 个）
- 管理员确认后系统自动生成 License
- 管理员确认后系统自动记账直接上级的首次奖励
- 管理员可拒绝并填写原因
- 所有佣金以"记账"形式记录

**Out of Scope:**
- 在线支付集成（H5 支付网关）

#### FR-7: 可售额度管理
代理/经销商可在"我的额度"中查看剩余可售额度和已销售记录。额度耗尽后可向管理员申请补购。

**Consequences (testable):**
- 代理初始额度 22 个，经销商 11 个，普通用户/888会员 0 个
- 额度销售后递减
- 额度为 0 时不可继续销售
- 额度耗尽后显示"申请补购"入口
- 管理员审核补购申请，确认后追加额度

### 4.3 邀请链与团队管理

**Description:** 系统记录完整的邀请关系链，每个用户可查看上级链条和下级团队树。

**Functional Requirements:**

#### FR-8: 邀请关系记录
系统在用户注册填写邀请码时，记录邀请关系。

**Consequences (testable):**
- 关系数据包含：上级用户 ID、下级用户 ID、邀请码、创建时间
- 关系不可删除或修改
- 支持通过关系表追溯完整链条

#### FR-9: 我的团队（下级树）
用户可查看以自己为根节点的下级团队树。

**Consequences (testable):**
- 展示每个直接下级的昵称、角色、注册时间、直接下级数量
- 支持逐层展开，不限层级深度
- 展示团队总人数统计

#### FR-10: 我的上级链条
用户可查看从自己向上追溯到根节点的完整上级链条。

**Consequences (testable):**
- 展示每一级上级的昵称、角色
- 根节点用户显示"无上级"
- 链条按层级排序

### 4.4 会员开通与额度销售

**Description:** 两个场景：(A) 代理/经销商用额度销售给客户（不产生佣金），(B) 下级自己充值（产生佣金）。

**Functional Requirements:**

#### FR-11: 额度销售（场景 A）
代理/经销商在"销售账号"页面录入客户邮箱（需邮箱验证），消耗 1 个可售额度，为客户开通 888 会员。**不产生佣金**。

**Consequences (testable):**
- 销售时校验可售额度 > 0
- 客户邮箱需邮箱验证
- 销售成功后额度减 1
- 客户成为 888 会员，生成 License
- 客户的上级 = 销售者（建立上下级关系）
- 不产生任何佣金记账
- 销售记录包含：销售人、客户邮箱、时间

#### FR-12: 推荐充值（场景 B）
下级自己充值 888/5000/10000，系统按规则记账直接上级的佣金。

**Consequences (testable):**
- 下级充 888：直接上级获得首次奖励（代理 488.4 元 / 经销商 355.2 元 / 普通用户或888会员 177.6 元）
- 下级充 5000：直接上级获得首次奖励（代理 2750 元 / 经销商 2000 元 / 普通用户或888会员 不产生佣金）
- 下级充 10000：直接上级获得首次奖励（代理 5500 元 / 经销商 4000 元 / 普通用户或888会员 不产生佣金）
- 普通用户/888会员推荐的人充 5000 或 10000 不产生佣金
- 后续收益：代理的直接下级经销商每次推荐他人充 888，代理获得 133.2 元/笔
- 所有佣金以"记账"形式记录

### 4.5 佣金计算与收益

**Description:** 系统根据角色和充值行为自动记账佣金。佣金分为首次奖励、后续收益、长期奖励三类。所有佣金以"记账"形式计入余额。所有参数均可在管理员后台配置。

**Functional Requirements:**

#### FR-13: 佣金规则配置（管理员可配）

**首次奖励（场景 B，下级充值时直接上级获得）：**

| 下级充值 | 上级是代理 (55%) | 上级是经销商 (40%) | 上级是普通用户/888会员 (20%) |
|---------|-----------------|------------------|--------------------------|
| 888 元 | 488.4 元 | 355.2 元 | 177.6 元 |
| 5000 元 | 2750 元 | 2000 元 | — |
| 10000 元 | 5500 元 | 4000 元 | — |

**后续收益（仅代理→经销商关系）：**
- 代理的直接下级经销商，每次推荐他人充 888（场景 B），代理获得 133.2 元/笔

**长期奖励（定期结算，仅限直接下级的全部收入总和 × 比例）：**

| 上级角色 | 下级角色 | 长期奖励比例 |
|---------|---------|------------|
| 代理 | 代理 | 5% |
| 代理 | 经销商 | 不适用（已由 133.2 元/笔替代） |
| 经销商 | 代理 | 4% |
| 经销商 | 经销商 | 4% |

**"全部收入"定义：** 直接下级获得的所有佣金总和，包括首次奖励 + 后续收益 + 长期奖励。

**Consequences (testable):**
- 每笔佣金记账精确到小数点后 2 位
- 佣金记账日志可追溯
- 管理员可在后台修改所有佣金比例和金额参数
- 参数修改后仅对新产生的业务生效

#### FR-14: 我的收益
用户可在"我的收益"页面查看收益明细。

**Consequences (testable):**
- 页面顶部展示：记账余额（待提现）、已提现总额、可用余额（= 记账余额 - 已冻结提现金额）
- "记账余额"使用独立的视觉样式（账本图标 + 说明文字"此为记账金额，需提交工单提现"）
- 收益明细按时间倒序排列
- 每条明细包含：金额、类型（首次奖励/后续收益/长期奖励/推荐返佣）、来源、时间
- 支持按类型筛选

#### FR-15: 长期奖励记账
系统按周期（默认按月）自动计算长期奖励。基于直接下级的全部收入总和 × 对应比例。

**Consequences (testable):**
- 仅计算直接下级的收入，不穿透更深层级
- 代理→经销商关系不适用长期奖励（已由 133.2 元/笔替代）
- 代理→代理关系：5%
- 经销商→代理关系：4%
- 经销商→经销商关系：4%
- 结算周期可由管理员配置（按月/按周/按天）
- 长期奖励以"记账"形式计入余额
- 幂等保护：同一周期不重复结算

### 4.6 License 对接

**Description:** 用户开通 888 会员后（含代理/经销商自带），系统生成 License Code。舆情系统通过调用本系统的验证接口确认 License 有效性。

**Functional Requirements:**

#### FR-16: License 生成
用户成为 888 会员/经销商/代理后，系统自动生成 License Code，绑定用户邮箱。

**Consequences (testable):**
- License 在充值确认后自动生成
- License 绑定用户邮箱，不可转让
- License 一次性使用，验证后标记为已激活
- License 格式为唯一字符串（含防篡改签名）
- 用户可在"我的"页面查看自己的 License

#### FR-17: License 验证接口
提供 API 供舆情系统验证 License 有效性。

**Consequences (testable):**
- 接口接收：License Code + 邮箱
- 验证逻辑：License 存在 + 邮箱匹配 + 未激活
- 验证通过：标记为已激活，返回成功
- 验证失败：返回具体错误（License 不存在 / 邮箱不匹配 / 已激活）
- 接口需鉴权（API Key 或 IP 白名单）

### 4.7 提现工单

**Consequences (testable):**

### 4.10 系统对接说明

**Description:** 本系统与足球舆情分析系统通过 License Code 对接，邮箱做用户关联。

**对接边界：**

| 职责 | 销售奖励系统（本项目） | 足球舆情分析系统 |
|------|---------------------|----------------|
| 用户注册 | **主** | 接收 License 验证 |
| 用户登录 | 各自独立（邮箱验证码） | 各自独立 |
| 邀请码 | **主** | 不涉及 |
| 角色管理 | **主** | 不涉及 |
| 888 会员开通 | **主**（生成 License） | 接收 License 验证 |
| License 验证 | **提供 API** | 调用 API |
| 舆情数据 | 不涉及 | **主** |
| 佣金/提现 | **主** | 不涉及 |

**对接流程：**
1. 用户在销售系统注册/充值 → 获得 License
2. 用户在舆情系统登录（邮箱验证码）→ 输入 License
3. 舆情系统调用销售系统 `POST /api/v1/license/verify` 验证
4. 验证通过 → 舆情系统开通高级功能
5. 未开通用户在舆情系统中引导跳转销售系统

---

### 4.11 跨领域非功能需求

#### NFR-1: 佣金记账幂等性
每笔佣金记账基于唯一业务 ID 进行幂等保护。

**Consequences (testable):**
- 同一笔业务不会产生重复佣金记账
- 幂等保护覆盖：首次奖励、后续收益、长期奖励

#### NFR-2: 审计日志
所有金额变动操作记录完整审计日志。

**Consequences (testable):**
- 审计日志包含：操作时间、操作人、操作类型、变更前后值、关联业务 ID
- 审计日志不可删除或修改

#### NFR-3: 邀请码安全性
邀请码包含防篡改签名。

**Consequences (testable):**
- 邀请码使用 HMAC-SHA256 签名
- 签名密钥仅服务端持有
- 被篡改的邀请码解析失败

#### NFR-4: License 安全性
License Code 防伪造、防转让。

**Consequences (testable):**
- License 绑定邮箱，验证时校验一致性
- License 一次性使用，激活后不可再用
- License 验证接口需鉴权

## 5. Non-Goals (Explicit)

- **不做在线支付**：充值均为线下 + 后台确认
- **不做在线提现**：提现通过工单 + 线下打款
- **不做角色升级**：各充值独立，888 会员充 5000 不退 888
- **不做复杂 BI 图表**：仅数字卡片
- **不做多端适配**：v1 仅 H5 + Web 管理后台
- **不做 SSO**：两系统各自独立登录，邮箱关联
- **不做社交功能**

- License 生成与验证接口
- 我的团队（下级树）+ 我的上级链条
- 我的收益（明细列表）
- 提现工单（申请 + 审核 + 打款确认）
- 管理员后台（用户管理 + 充值审核 + 数据看板 + 参数配置）
- H5 站内/邮件消息通知

- 在线支付集成 → v2
- 在线提现 → v2
- 复杂数据图表 → v2
- 原生 App / 支付宝端 → v2
- 多语言支持 → v2
