FAIR

The PRD delivers strong, testable execution specs for a well-understood distribution system: every FR has concrete consequences, the glossary is comprehensive, and the Vision is earned and specific. What holds up is downstream build readiness. What is at risk is decision transparency — the PRD surfaces no unresolved tensions, carries no inline assumption tags, and lacks callouts where trade-offs were made.

The code audit confirms 11 of 12 previous-round fixes are correctly implemented, but reveals one critical production-blocking issue: the alembic migration chain is out of sync with the ORM models. A fresh alembic upgrade head would create tables that don't match the current model schema, causing application failures on first query. Three high-severity code gaps also remain: notification emails never sent (FR-24 violation), missing referral_relationships table (FR-8 violation), and scheduler config key mismatch (S6 regression).

Code implementation audit

review-code-audit.md

11/12 previous-round fixes verified. S6 (scheduler) partially broken due to config key mismatch between settlement_cycle_days (queried) vs settlement_cycle (auto-initialized).

critical

Alembic migrations out of sync with ORM models

backend/alembic/versions/003*.py
Migration 003 creates recharges/invite_codes tables. Current ORM expects payments/referral_codes/quota_replenishments. Fresh alembic upgrade head fails on first ORM query. 6 table/column/enum mismatches identified.
Fix: Create new migration (008) that drops old tables and creates current schema. OR refactor migrations 003-007 into a single clean migration.
high

Notification emails never sent (FR-24 violation)

notification_service.py:112
_send_notification_email() defined but never invoked. All 5 trigger scenarios (subordinate payment, commission credited, ticket change, payment approved, seed user created) only write DB records — no email delivered.
Fix: Call _send_notification_email() from send() or each notify_* static method.
high

Missing referral_relationships table (FR-8 violation, S9)

No table exists
PRD requires immutable relationship records (upstream ID, downstream ID, referral code, payment ID, timestamp). Currently only parent_id FK on User — relationships are mutable, untraceable to payment, and 888 commission-only relationships have no formal tracking.
Fix: Create referral_relationships table + migration + update payment_service to write on relationship establishment.
high

Scheduler config key mismatch (S6 regression)

scheduler_service.py + system_config_service.py
Queries settlement_cycle_days (never auto-initialized), but DEFAULT_CONFIGS initializes settlement_cycle. Scheduler always falls back to 30 days. Dead _get_cron_trigger() function correctly uses the existing key but is never called.
Fix: Add settlement_cycle_days to DEFAULT_CONFIGS or route scheduler through _get_cron_trigger().
medium

PayPage redirect whitelist hardcoded

PayPage.tsx:10
['localhost', '127.0.0.1'] — production domains cannot be added without code change.
Fix: Make configurable via env var or build-time injection.
medium

Unused "refunded" payment status

models/payment.py:31
Enum includes "refunded" but PRD Non-Goals explicitly state "不做退款".
Fix: Remove enum value or add clarifying comment.

NFR compliance audit

review-nfr.md

3 of 5 NFRs fully compliant (NFR-1, NFR-3, NFR-4). NFR-2 and NFR-5 are partial with documented gaps.

medium

NFR-2: Audit log lacks immutability protection

audit_log.py
PRD requires logs "不可删除或修改". Zero protection: no ORM event listener, no DB trigger, no append-only configuration. Any user with write access can modify or delete records.
Fix: Add ORM-level event listener preventing UPDATE/DELETE on AuditLog; for production, add DB trigger.
medium

NFR-5: Callback uses custom HMAC, not WeChat/Alipay signatures

payments.py:84-99
Internal HMAC-SHA256 replaces official WeChat V3 RSA-SHA256 or Alipay RSA2 verification. TODO comment acknowledges demo status. Adequate for mock/test; must be replaced before production.
Fix: Replace with official payment SDK signature verification before production deployment.
