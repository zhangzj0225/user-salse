# Sprint Change Proposal

## 1. Issue Summary

### 变更触发原因

在准备与舆情系统（业务系统）进行真实对接时，发现当前分销模型存在根本性矛盾：

1. **用户体系矛盾**：当前模型假设终端用户在分销系统注册，但实际上业务系统开放注册，两系统用户独立。分销系统用户生成邀请码去邀请业务系统用户注册的场景不存在。
2. **邀请码定位错误**：邀请码用于"注册时建立上下级关系"，但终端用户不在分销系统注册，邀请码的核心用途消失。
3. **支付模式缺失**：当前 PRD 明确 Non-Goal"不做在线支付"，但业务系统需要用户能自主充值 888 购买会员，需要一个在线支付闭环。
4. **管理员初始化缺失**：取消开放注册后，首批代理/经销商需要一个创建入口。

### 变更范围

本次变更为 **Major 级别**，涉及 PRD、架构、Epics 的结构性调整：

- PRD：FR-1~FR-4 重写，FR-17 修改，FR-20 增强，新增 FR-25(在线支付)
- 架构：用户模型简化，新增支付模块，License 验证简化，推荐码替代邀请码
- Epics：现有 Story 修改 + 新增支付相关 Story

## 4. Detailed Change Proposals

### 4.1 PRD 变更

#### FR-1: 用户登录（重写）

**OLD:**
> 用户通过邮箱验证码完成注册与登录。`/login` 与 `/register` 共用邮箱验证码机制。`/login`：邮箱已存在则登录既有用户，不存在则冷启动创建根用户。`/register`：邮箱已注册时提示"邮箱已注册"。

**NEW:**
> 分销系统仅面向代理、经销商和管理员。`/login` 使用邮箱验证码登录已注册用户，不创建新用户。终端用户（888会员）不在分销系统注册，仅在业务系统注册。代理/经销商的创建通过两条路径：(1) 超管在管理后台直接创建种子用户；(2) 通过支付小闭环在线支付 5000/10000 后自动创建账号。

**Consequences:**
- `/login` 仅登录，不创建用户（邮箱不存在时提示"用户不存在"）
- 不再提供 `/register` 页面给终端用户
- 代理/经销商通过支付或超管创建后，用邮箱验证码登录分销系统
- 验证码 5 分钟有效

#### FR-2: 邮箱即账号标识（重写）

**OLD:**
> 邮箱是用户在本系统的唯一账号标识，也是两系统关联的桥梁。

**NEW:**
> 邮箱是代理/经销商/管理员在分销系统的唯一账号标识。终端用户（888会员）在业务系统有独立的账号体系，不在分销系统注册。两系统通过 License Code 对接，不共享用户体系。

#### FR-3: 推荐码（重写，原"邀请码填写与关系建立"）

**OLD:**
> 用户在 `/register` 注册时必须填写邀请码。邀请码为统一类型，一次性使用。

**NEW:**
> 推荐码是代理/经销商的身份标识码，用于支付时标识推荐人。推荐码为持久码，每人 1 个，可重复使用。用户在支付小闭环支付 888/5000/10000 时可选填推荐码。填了推荐码 → 计算佣金；不填 → 无佣金，平台收全款。

**Consequences:**
- 推荐码持久有效，不消耗
- 每个代理/经销商 1 个固定推荐码
- 推荐码包含 HMAC-SHA256 防篡改签名
- 支付时选填，不强制
- 推荐码不建立"注册关系"，只建立"佣金归属关系"

#### FR-4: 推荐码生成（重写）

**OLD:**
> 注册成功后，系统为用户生成个人邀请码。所有角色均可生成。可生成多个未使用的。

**NEW:**
> 代理/经销商创建时（超管创建或支付成功），系统自动生成 1 个持久推荐码。管理员不生成推荐码。推荐码在用户生命周期内不变。

#### FR-11: 额度销售（修改）

**OLD:**
> 代理/经销商录入客户邮箱（需邮箱验证），消耗 1 个额度，为客户开通 888 会员，生成 License。客户的上级 = 销售者。

**NEW:**
> 代理/经销商在后台点击"生成 License"，消耗 1 个额度，生成一个未激活的 License Code。License 生成时不绑定任何用户信息。代理将 License Code 交付给客户（卖或送），客户在业务系统输入 Code 激活。激活时业务系统调用验证 API，分销系统将 License 标记为已激活并绑定业务系统用户标识。不产生佣金。

**Consequences:**
- License 3 种状态：`unused`（未激活）→ `active`（已激活）
- 生成时不绑定邮箱
- 激活时绑定业务系统传来的用户标识（`activated_user_ref` 字段）
- 不产生佣金

#### FR-12: 推荐充值（修改）

**OLD:**
> 下级充 888：代理 488.4 / 经销商 355.2 / 普通用户或888会员 177.6
> 下级充 5000：代理 2750 / 经销商 2000 / 普通用户不产生
> 下级充 10000：代理 5500 / 经销商 4000 / 普通用户不产生

**NEW:**
> 用户通过支付小闭环支付 888/5000/10000，如填了推荐码则计算佣金。无推荐码则无佣金。

| 支付金额 | 推荐人=代理 (55%) | 推荐人=经销商 (40%) | 无推荐码 |
|---------|:---:|:---:|:---:|
| 888 | 488.4 | 355.2 | 无佣金 |
| 5000 | 2750 | 2000 | 无佣金 |
| 10000 | 5500 | 4000 | 无佣金 |

#### FR-13: 佣金规则配置（修改）

**OLD:** 3 档（55%/40%/20%）
**NEW:** 2 档（55%/40%），新增"无推荐码=无佣金"规则

首次奖励矩阵：

| 支付金额 | 推荐人=代理 (55%) | 推荐人=经销商 (40%) | 无推荐码 |
|---------|:---:|:---:|:---:|
| 888 | 488.4 | 355.2 | — |
| 5000 | 2750 | 2000 | — |
| 10000 | 5500 | 4000 | — |

后续收益（仅代理→经销商链）：经销商的推荐码用户支付 888 → 经销商的上级代理获 133.2 元/笔

长期奖励：

| 上级 | 下级 | 比例 |
|------|------|------|
| 代理 | 代理 | 5% |
| 代理 | 经销商 | 不适用（133.2 替代） |
| 经销商 | 代理 | 4% |
| 经销商 | 经销商 | 4% |

"全部收入"定义：直接下级的首次奖励 + 后续收益 + 长期奖励总和（额度销售不产生收入，不计入）

#### FR-17: License 验证接口（修改）

**OLD:**
> 接口接收：License Code + 邮箱。验证逻辑：License 存在 + 邮箱匹配 + 未激活。

**NEW:**
> 接口接收：License Code + 业务系统用户标识（可选）。验证逻辑：License 存在 + 未激活。验证通过后标记为已激活，绑定业务系统用户标识。

**Consequences:**
- 不校验邮箱（两系统用户独立）
- License Code + 未激活 → 验证通过
- 验证通过后绑定 `activated_user_ref`（业务系统传来的用户标识）
- 已激活的 License 再次验证返回"已激活"状态
- 接口需 API Key 鉴权

#### FR-20: 用户管理（增强）

**OLD:**
> 管理员可查看所有注册用户列表。

**NEW:**
> 管理员可查看所有用户列表，并可直接创建种子用户（代理/经销商）。

**新增子功能 — 创建种子用户：**
- 超管指定邮箱 + 角色（代理/经销商）→ 系统直接创建用户
- 跳过支付流程，自动分配额度（代理 22 / 经销商 11）
- 自动生成个人 License + 推荐码
- 种子用户用邮箱验证码登录分销系统
- 可选填推荐码建立上级关系

#### FR-25: 在线支付小闭环（新增）

**Description:**
分销系统提供独立的支付页面，支持 888/5000/10000 三档在线支付。支付成功后自动生成 License（888）或创建账号+分配额度+生成License（5000/10000），并计算佣金。

**Consequences:**
- 支付页面独立部署，URL 格式：`/pay?amount={888|5000|10000}&referral={推荐码}&redirect={回跳URL}`
- 支付方式：微信扫码 / 支付宝扫码
- 支付信息收集：邮箱（必填）、推荐码（选填）
- 888 支付：生成 License → 邮件发送 + 页面展示 → 引导回业务系统激活
- 5000 支付：创建经销商账号 + 11 额度 + License + 推荐码 → 计算佣金
- 10000 支付：创建代理账号 + 22 额度 + License + 推荐码 → 计算佣金
- 支付回调验签 + 幂等保护
- `redirect` 参数白名单校验
- 支付成功后 License Code 通过 URL 参数回传给业务系统

### 4.2 架构变更

#### 用户模型简化

```
Before:                          After:
┌─────────────────┐              ┌─────────────────┐
│ User            │              │ User            │
│ - email (unique)│              │ - email (unique)│
│ - role:         │              │ - role:         │
│   user          │              │   agent         │
│   member (888)  │   ────────>  │   distributor   │
│   distributor   │              │ - referral_code │
│   agent         │              │ - quota         │
│ - invite_code   │              │ - parent_id     │
│ - parent_id     │              └─────────────────┘
│ - quota         │
└─────────────────┘              终端用户不在分销系统
```

#### 推荐码模型（替代邀请码）

```
Before (InviteCode):              After (ReferralCode):
┌──────────────────┐              ┌──────────────────┐
│ code (unique)    │              │ code (unique)    │
│ generator_id     │              │ user_id (1:1)    │
│ used_by (nullable)│             │ persistent       │
│ used_at (nullable)│             │ never consumed   │
│ one-time use     │              └──────────────────┘
└──────────────────┘              每人1个，持久有效
```

#### License 模型调整

```
新增字段:
  - activated_user_ref (String, nullable)  — 业务系统用户标识
  - payment_id (FK, nullable)              — 关联支付记录

状态流转:
  unused → active (验证通过时)
  (不再有 expired 状态，不支持的退款)
```

#### 新增支付模块

```
payment_service.py
├── create_payment(amount, email, referral_code, redirect_url)
├── handle_callback(payment_data)  — 验签 + 幂等
├── on_payment_success(payment)
│   ├── 888: 生成 License + 计算佣金
│   ├── 5000: 创建经销商 + 额度 + License + 佣金
│   └── 10000: 创建代理 + 额度 + License + 佣金
└── generate_license_for_payment(payment, email)
```

#### 对接接口

```
业务系统 → 分销系统:
  POST /api/v1/license/verify
    Request: { license_code, user_ref? }
    Response: { valid, status, activated_user_ref? }

  GET /pay?amount=888&referral=XXX&redirect=https://业务系统/activate
    (浏览器跳转，非API)
```
