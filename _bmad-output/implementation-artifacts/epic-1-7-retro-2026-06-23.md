## 关键经验教训

### 1. PRD 中途大改的代价与收益

**情况**: 项目进行到 Epic 3 完成后，因业务系统对接需求明确，PRD 从 v1 大幅改为 v2

**代价**:
- 91 个文件重写，+6558/-4719 行
- 邀请码→推荐码、3 档佣金→2 档、注册流程删除、License 不绑定邮箱
- 大量 E2E 测试需要适配

**收益**:
- 用户体系独立（分销系统 vs 业务系统），解耦正确
- 佣金模型简化（去掉 20% 档），逻辑更清晰
- License 激活时绑定业务系统用户，更符合实际场景

**教训**: 早期与业务系统的对接边界应在 PRD v1 阶段就明确。如果 v1 时就问"终端用户在哪里注册？"，可以避免 v2 的大规模重构。

### 2. 代码审查发现的安全问题

**后端 CRITICAL（3 项）**:
- 支付回调无认证 → 增加 HMAC-SHA256 签名
- 事务中途 commit → 改为 flush-only
- pending_user_key 从未赋值 → 使用 email hash

**前端 CRITICAL（1 项）**:
- 开放重定向漏洞 → redirect 白名单校验

**教训**: 支付相关的安全审查应在设计阶段就介入，不应等到代码审查才发现。签名认证、事务边界、重定向校验是支付场景的"必检项"。

### 3. 外部补丁的价值

**情况**: 代码 push 后在外部平台 review，收到补丁 `bdb4c0f`

**补丁修复了 3 个我遗留的问题**:
- commission savepoint（我只改了注释没改代码）
- 回调密钥独立（我复用了 LICENSE_API_KEY）
- redirect 后端校验（我只做了前端白名单）

**教训**: 自己的代码审查容易有盲区。外部 review 提供了不同视角，补充了遗漏。应建立"自我审查 + 外部审查"的双重机制。

### 4. E2E 测试的演进

**阶段**:
1. 初始：chain_e2e.py（19 个用例）— 基础链路验证
2. 扩展：e2e_extended.py（50 个）+ e2e_gap_cover.py（31 个）— 覆盖矩阵
3. 规则：e2e_commission_rules.py（19 个）— 佣金边界条件
4. 全流程：e2e_full_flow.py（29 个）— 端到端用户旅程
5. 外部补丁：+23 个新场景 + 6 项断言改进

**教训**: E2E 测试应从项目早期就建立，随功能迭代增量扩展。断言质量比数量更重要（"vacuous truth" → 精确 DB 查询）。

### 5. Mock 优先策略的有效性

**情况**: 邮箱验证码先用 Mock（固定 123456），后期才实现真实 SMTP

**收益**:
- 开发阶段不依赖邮件服务，快速迭代
- 后期对接 SMTP 时，测试框架已完善，可以真实邮箱验证
- 真实邮箱 E2E（QQ 邮箱自动登录 + 验证码读取）7/7 通过

**教训**: Mock 优先是正确的策略，但必须在 config 层有生产环境校验（SEC-2），防止 Mock 进入生产。
