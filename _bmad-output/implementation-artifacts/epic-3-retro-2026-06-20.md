## What Went Well

### 1. Test-Driven Development Pattern

Every story delivered with comprehensive tests before code review. Tests served as living documentation of acceptance criteria.

**Example:** Story 3.11 (License) delivered 27 tests covering generation, verification, activation, tamper detection, and API authentication.

### 2. Iterative Code Review Process

Three review rounds caught progressively deeper issues:
- **Round 1 (self):** N+1 queries, missing response_models, schema mismatches
- **Round 2 (peer):** P0 concurrency bugs (F1 double-spend, F2 FK), role downgrade (BH-3), security (SEC-1)
- **Round 3 (self):** Previous fix regressions (F4 savepoint invalid, fake tests), fail-closed default

### 3. Shared Balance Calculation Function

Peer reviewer introduced `calculate_balance_summary()` to eliminate formula drift between earnings and withdrawal services. This became the single source of truth for balance computation.

### 4. DB-Level TOCTOU Prevention (D5)

`pending_user_key` nullable column + UNIQUE constraint provided cross-database compatible defense against concurrent pending recharge creation. More portable than partial indexes.

### 5. MySQL Concurrent Test Infrastructure

`docker-compose.test.yml` + `test_concurrent_mysql.py` established infrastructure for verifying row locks actually work (SQLite `with_for_update()` is no-op). 3 real multi-threaded tests cover withdrawal overspend, double role change, and duplicate license activation.

## What Didn't Go Well

### 1. Concurrency Safety Gaps in Initial Implementation

Multiple P0/P1 concurrency bugs were introduced and only caught in peer review:

| Bug | Severity | Root Cause |
|-----|----------|------------|
| F1 double-spend | P0 | `with_for_update` only locked Recharge row, not User row |
| BH-3 role downgrade | P1 | `user.role = AMOUNT_ROLE_MAP[amount]` overwrote higher roles |
| F5 concurrent pending | P1 | No check for existing pending recharge |
| F6 withdrawal lock | P1 | Locked empty pending set instead of User row |

**Lesson:** Concurrency design must be reviewed by a second pair of eyes. The original developer's mental model of "I added with_for_update" was insufficient — the lock scope was wrong.

### 2. Savepoint Approach Was Invalid (D1)

Round 2 attempted to use `begin_nested()` for idempotent commission recording. Round 3 discovered this was fundamentally broken in SQLAlchemy 2.0 — `flush` hitting UNIQUE constraint puts the session into rollback-pending state that savepoints cannot recover from.

**Lesson:** Don't assume ORM transaction behavior matches expectations. Test the actual failure path, not just the happy path.

### 3. Fake Test That Passed (D2)

Round 2's test for idempotent degradation never reached the savepoint code path — it hit the `existing` check first and returned `None`. The test passed but validated nothing.

**Lesson:** Always verify test effectiveness by temporarily breaking the code under test and confirming the test fails.

### 4. Security Default Was Fail-Open (D3)

Round 2's SEC-1 implementation defaulted `ENV=dev`, meaning production deployments that forgot to set `ENV=production` would silently use default secret keys. Round 3 fixed to fail-closed default.

**Lesson:** Security defaults should be fail-closed. Explicit opt-in for relaxed behavior, not opt-out.

### 5. Balance Formula Drift Between Services

Earnings service hardcoded `withdrawn=0` when withdrawal wasn't implemented yet. When withdrawal was added in a later story, the earnings service wasn't updated, causing inconsistent available balance displayed to users.

**Lesson:** When implementing a service that depends on future functionality, use a shared calculation function from the start, even if it returns placeholder values.

## Key Insights

### 1. Concurrency Defense in Depth

The final architecture uses three layers of concurrency protection:

```
Layer 1: Application check (query existing pending)
Layer 2: Row lock (with_for_update on User/Recharge/License)
Layer 3: DB constraint (UNIQUE on pending_user_key, business_id)
```

No single layer is sufficient. The application check is TOCTOU-vulnerable, the row lock is no-op on SQLite, and the DB constraint only catches the specific case. Together they provide robust protection.

### 2. Role Management: Highest-Wins, Not Last-Wins

The `_higher_role()` function ensures充值 only upgrades, never downgrades. This aligns with PRD "各充值独立不互斥" — an agent who buys 888 again should not lose agent status.

### 3. Audit Log Coverage

All state-changing operations now write audit logs:
- `recharge_approve` / `recharge_reject`
- `withdrawal_create` / `ticket_approve` / `ticket_reject`
- `commission_create`
- `sale_account`

### 4. License Code Design

The 4-segment format (`Base62(uid).email_hash8.nonce8.hmac16`) provides:
- Tamper detection (HMAC)
- Email binding (hash verification before DB lookup)
- Uniqueness (random nonce)
- Defense in depth (user_id cross-verification)
- Log safety (only first 8 chars logged)
